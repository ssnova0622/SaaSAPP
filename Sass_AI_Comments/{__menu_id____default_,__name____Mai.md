### After choosing `select_timeslot`, how does the flow continue?

Short answer: selecting the `select_timeslot` action starts a small booking state machine (FSM). From that point on, the “next options” are not other menu nodes — they are dynamic prompts generated by the FSM, which guides the user through picking a professional, picking a time slot, confirming, and providing name/phone. The FSM uses the WhatsApp session context to remember progress and expects simple numeric replies like `1`, `2`, etc.

---

### Where this is implemented

- Action dispatch: `_run_action(...)` maps your menu node’s `action_id` to a handler.
  - Your `"action_id": "select_timeslot"` is treated as a legacy alias and mapped to `"salon.select_timeslot"` inside `_run_action` via `_legacy_to_action_id(...)`.
  - Handler: `_action_select_timeslot(tenant, locale)` (see `app/routers/whatsapp.py`).
- The interactive booking steps are handled by: `_handle_timeslot_fsm(tenant, phone, user_input, tree)`.
- Session helpers: `_get_session`, `_save_session`, `_reset_session_to_root` manage `last_node` and `ctx` in Mongo.

---

### Step-by-step flow after `select_timeslot`

1. User selects the “Book appointment” option in your submenu, which points to the `action` node `{ id: "book", type: "action", action_id: "select_timeslot" }`.
2. The backend calls `_run_action(...)` → resolves to `salon.select_timeslot` → calls `_action_select_timeslot(...)`.
3. `_action_select_timeslot(...)` returns an initial message with available slots. If there are multiple professionals configured, the system first asks to choose a professional. Otherwise it lists slots for the first professional with availability.
   - Example prompt (multiple professionals):
     - "Choose a professional:\n1) Alice\n2) Bob"
   - Example prompt (single/first available professional):
     - "Available time slots with Alice:\n1) 10:00\n2) 11:00\n...\nReply with a number to choose a slot."
4. The user now replies with a number, and the FSM takes over via `_handle_timeslot_fsm(...)` based on `ctx.mode`:
   - Modes and transitions:
     - `select_prof` → expects a number to select a professional, then shows that professional’s slots and moves to `select_slot`.
     - `select_slot` → expects a number to pick a time; then moves to `confirm_booking` and asks:
       - "Confirm booking for 11:00?\n1) Yes\n2) No"
     - `confirm_booking` → if `1` (Yes), moves to `ask_name`; otherwise returns "Okay, not booked." and resets to menu root.
     - `ask_name` → expects a name or `skip`; then moves to `ask_phone`.
     - `ask_phone` → expects a phone number or `use wa` (uses the WhatsApp number). Then it calls `Storage.create_appointment(...)` and replies with a confirmation like:
       - "Booked! Appointment ID ABC123 at 11:00."
     - After booking or cancellation, the session is reset to the menu root.

---

### Where the “next option” comes from

- It’s generated by the FSM prompts, not by your static menu tree after the action node.
- The next screens/options the user sees are the text prompts returned by `_action_select_timeslot(...)` and `_handle_timeslot_fsm(...)` according to the current `ctx.mode`.
- Numeric replies are parsed by `_extract_choice` and the local helpers inside the FSM to select items like `1)`, `2)`, etc.

### Key functions to check in code

- `_action_select_timeslot(...)` — initial prompt with professionals or slots.
- `_start_timeslot_flow(...)` — used to initialize/continue the slot selection prompts.
- `_handle_timeslot_fsm(...)` — processes user replies across modes: `select_prof` → `select_slot` → `confirm_booking` → `ask_name` → `ask_phone` → booking.
- `_run_action(...)` — how the action is dispatched (and how legacy IDs are handled).

### Can I redirect back to a menu after booking or add more steps?

- After booking or cancel, the code calls `_reset_session_to_root(...)` so the user lands back at the root menu.
- If you want a custom post-booking flow (e.g., show another submenu or send a follow-up action), you can:
  - Adjust the message returned after booking to include a clear instruction like “Reply 1 to go back to menu” and add a trigger for that.
  - Or add a new step in the FSM after `ask_phone` to send another prompt/options before resetting.
