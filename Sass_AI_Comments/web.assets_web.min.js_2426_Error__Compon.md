### Findings in ofh_sale_order_sap/static/src/js
I reviewed the three JS files loaded by the module:
- open_sap_dialog.js: registers an action that opens a dialog (looks fine in structure).
- sap_dialog_content.js: onConfirm calls a model method via ORM, then dispatches a window event and closes the dialog.
- form_refresh.js: patches FormController to listen for the window event and reload the current record.

These pieces can easily produce the “Component is destroyed” error if the controller unmounts while an awaited load/refresh is still in flight (typical in OWL when a dialog or action triggers another view update).

### What’s risky in the current code
1. form_refresh.js reload logic
   ```javascript
   async _onSapRefresh(ev) {
       if (ev.detail.recordId === this.model.root.resId) {
           if (this.isDestroyed) return;
           await this.model.root.load();
       }
   }
   ```
   - If the controller is unmounted after `load()` begins (e.g., navigation, action change), the model’s async path will still resolve and try to update the destroyed component → “Component is destroyed”.
   - There is no guard after the await, and no cancellation/debouncing.

2. sap_dialog_content.js dispatch order
   ```javascript
   await this.env.services.orm.call(modelName, method, [[recordId]]);
   window.dispatchEvent(new CustomEvent("sap:refresh", { detail: { recordId } }));
   this.props.close();
   ```
   - The event fires while the dialog is still mounted and possibly while the parent view might be transitioning. If the refresh causes a view change (or a quick re-render) at the same time the dialog is closing, you get overlapping lifecycles.

### Minimal hardening to prevent the error
Apply these defensive adjustments.

1) In sap_dialog_content.js: close first, then dispatch in the next microtask
```javascript
// after successful ORM call
await this.env.services.orm.call(modelName, method, [[recordId]]);
// Close the dialog before telling forms to refresh
this.props.close();
// Dispatch after a microtask so closing/unmounting settles first
Promise.resolve().then(() => {
    window.dispatchEvent(new CustomEvent("sap:refresh", { detail: { recordId } }));
});
```

2) In form_refresh.js: strengthen guards, debounce, and re-check after await
```javascript
/** @odoo-module **/
import { patch } from "@web/core/utils/patch";
import { FormController } from "@web/views/form/form_controller";

patch(FormController.prototype, {
    setup() {
        super.setup();
        this._sapRefreshHandler = this._onSapRefresh.bind(this);
        window.addEventListener("sap:refresh", this._sapRefreshHandler);
        this.__sap_refresh_inflight = null;
    },
    willUnmount() {
        super.willUnmount();
        window.removeEventListener("sap:refresh", this._sapRefreshHandler);
        this.__sap_refresh_inflight = null;
    },
    async _onSapRefresh(ev) {
        // Basic sanity checks
        if (this.isDestroyed || !this.model || !this.model.root) return;
        const currentId = this.model.root.resId;
        if (!currentId || ev.detail.recordId !== currentId) return;

        // Debounce/collapse multiple refreshes
        if (this.__sap_refresh_inflight) return;
        const token = {};
        this.__sap_refresh_inflight = token;

        try {
            await this.model.root.load();
        } catch (e) {
            // Ignore lifecycle-related errors; rethrow others if needed
            // (In minified builds, message may be generic; keep silent for UX.)
        } finally {
            if (this.__sap_refresh_inflight === token) {
                this.__sap_refresh_inflight = null;
            }
        }

        // If we got unmounted while awaiting, do nothing further
        if (this.isDestroyed) return;
    },
});
```

This keeps the refresh from running at all if the controller is already destroyed; collapses duplicate events; and ensures we don’t proceed after unmount.

### Additional notes
- If the action opening the dialog is triggered from a list/tree view, it may be the list (not the form) that’s reloading in your stack trace (`_loadRecords`, `_loadData`). If you also want lists to refresh, create a similar safe patch for ListController (or better, avoid using global window events and use the bus/useBus hook to scope listeners to the mounted component).
- Prefer the framework bus over a global `window` event:
  - In OWL, `useBus` from `@web/core/utils/hooks` can listen on `env.bus` with automatic cleanup on unmount.
- If any other button returns an `ir.actions.act_url` with `target: 'self'` (e.g., for downloads), change to `target: 'new'` to avoid immediate navigation that unmounts the view while promises resolve.
