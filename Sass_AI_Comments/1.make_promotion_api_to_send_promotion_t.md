### Plan for the requested features

Below is a concrete, step‑by‑step plan to add the 5 capabilities to the AI Scheduler (Mongo‑backed FastAPI app under `ai_scheduler/`). The plan is organized by features, with data models, endpoints, background jobs, and delivery details. We’ll keep the MongoDB usage consistent with the existing `storage_mongo` and reuse the Twilio webhook path for outbound messages where relevant.

---

### 0) Cross‑cutting foundations (applies to all features)
- Authentication/authorization (minimal):
  - Add a basic API key or bearer token middleware for admin endpoints (`X-API-Key` from env for now).
  - Roles: `admin` (tenant owners) vs `ops` (can send promotions) vs `viewer`.
- Background job framework:
  - Add APScheduler (or `fastapi-utils`/`BackgroundTasks`) for scheduled tasks and delayed follow‑ups.
  - One process scheduler to run periodic jobs: daily invoice, follow‑up dispatcher, retention reminders.
- Messaging service abstraction:
  - Create `ai_scheduler/services/messaging.py` with functions: `send_whatsapp_text(to, message)`, `send_whatsapp_media(to, media_url, caption)`, `send_sms(...)` (optional), `send_email(...)` (optional). Implementation can use Twilio and SMTP; keep no‑op/dev mode for local.
- Email configuration (optional but recommended for invoices):
  - `settings.py`: `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASSWORD`, `SMTP_FROM`.

### 1) Promotion API — send promotions to customers

Data model (Mongo collections):
- `customers` (reuse from `app/` if shared, else create under `ai_scheduler`): `{ tenant, phone, name, tags:[], last_seen_at, total_bookings }`
- `promotions`: `{ _id, tenant, name, channel: 'whatsapp'|'sms'|'email', message, media_url?, audience: { type: 'all'|'tags'|'custom', tags?:[], phones?:[] }, schedule_at?, created_at, status: 'draft'|'scheduled'|'running'|'completed'|'canceled' }`
- `promotion_logs`: `{ promotion_id, tenant, to, channel, status: 'queued'|'sent'|'failed', error?, sent_at }`

Endpoints (under new router `ai_scheduler/routers/promotions.py`):
- `POST /v1/tenants/{tenant}/promotions` — create promotion (draft or scheduled).
- `GET /v1/tenants/{tenant}/promotions` — list promotions.
- `GET /v1/tenants/{tenant}/promotions/{id}` — get details + aggregate stats.
- `PUT /v1/tenants/{tenant}/promotions/{id}` — update if not started.
- `POST /v1/tenants/{tenant}/promotions/{id}/send` — trigger immediate send (or schedule if `schedule_at`).
- `GET /v1/tenants/{tenant}/promotions/{id}/logs` — paginated delivery logs.

Logic:
- Audience resolution: based on `audience.type`.
- Rate limiting: throttle to Twilio limits; batch in chunks (e.g., 20/second).
- Delivery via `messaging.py` with retries/backoff; store one `promotion_logs` doc per recipient.
- Realtime: broadcast summary updates to `/ws/{tenant}` (`promotion.started`, `promotion.progress`, `promotion.completed`).

### 2) Admin screen — CRUD operations

Scope: lightweight admin for tenants, professionals, customers, promotions, and appointments.

Approach options:
- A) Minimal server‑rendered views using FastAPI Jinja templates (similar to `app/`), static assets, and the existing APIs.
- B) API‑only plus a simple frontend page (HTML/HTMX) served by FastAPI.

MVP Plan (A):
- New router `ai_scheduler/routers/admin_ui.py` with pages:
  - `GET /admin` — dashboard (per tenant choice).
  - `GET /admin/{tenant}/professionals`, create/update/delete forms.
  - `GET /admin/{tenant}/promotions`, creation & send page.
  - `GET /admin/{tenant}/customers`, create/update/import CSV.
  - `GET /admin/{tenant}/appointments` (list/cancel).
- Reuse API endpoints for actions; pages call them via forms.
- Auth gate: API key check middleware for paths under `/admin`.

If you prefer a frontend-less API approach, we keep only REST endpoints and provide a Postman collection.

### 3) Daily invoice/report PDF per tenant

Goal: Every day (e.g., 19:30 local tenant time), send a PDF summary for the day to the tenant owner.

Data/Config:
- Extend `tenants` with report settings: `{ owner_email, owner_phone?, tz, invoice_delivery: 'email'|'whatsapp'|'both' }`.

PDF generation:
- Add `ai_scheduler/services/reports.py` using `reportlab` or `WeasyPrint` to render:
  - Header with tenant, date, totals.
  - Table of appointments: time, professional, customer (masked), price, status.
  - Totals: revenue, count, cancellations.
- Store file:
  - Option 1: Generate on the fly and attach to email/whatsapp (S3/MinIO upload optional).
  - Option 2: Save to disk or GridFS with metadata: `{ tenant, date, url/path }`.

Delivery:
- Email via SMTP (`send_email` with PDF attachment) and/or WhatsApp media message (Twilio supports media URLs; if attachment needed, upload to a public URL or serve a signed URL endpoint like `/v1/tenants/{tenant}/reports/{date}.pdf`).

Scheduler:
- APScheduler daily job per tenant using configured TZ:
  - Query appointments for `date=today` and compute summary (reuse existing analytics logic + filter by date range).
  - Generate PDF and deliver per settings.
  - Log to `reports` collection: `{ tenant, date, sent_via, status, error? }`.

Endpoints:
- `POST /v1/tenants/{tenant}/reports/daily/run` — manual trigger for a date.
- `GET /v1/tenants/{tenant}/reports/daily/{yyyy-mm-dd}` — fetch metadata/download link.

### 4) Follow‑up logic for appointment customers

Use cases:
- After appointment is booked: send confirmation immediately.
- Reminder before the appointment (e.g., T‑24h / T‑2h).
- Post‑visit follow‑up (feedback link, rebooking suggestion at N days).
- If appointment canceled/no‑show: send a recovery message (optional).

Data model:
- `followups`: `{ _id, tenant, appointment_id, to, type: 'confirm'|'reminder'|'post'|'recovery', run_at, status: 'scheduled'|'sent'|'failed'|'canceled', last_error? }`

Workflow:
- On `create_appointment`, enqueue followups:
  - `confirm` at now.
  - `reminder_24h` (if time > 24h ahead).
  - `reminder_2h` (if time > 2h ahead).
  - `post` at appointment_time + 4h (or next morning) with feedback link.
- On `cancel_appointment`, cancel future followups for that appointment and optionally enqueue `recovery`.
- Dispatcher job (every minute): send due followups via `messaging.py`; update status and log failures with retry policy.

Config:
- Per tenant follow‑up preferences in `tenants` doc: which followups enabled, timing offsets, message templates (with placeholders: `{{customer_name}}`, `{{professional}}`, `{{time}}`).

### 5) Customer retention logic

Objectives:
- Identify customers who haven’t visited for X days (e.g., 30/60/90) and target them with win‑back campaigns.
- Compute simple retention score.

Data & metrics (extend `customers`):
- Maintain `last_visit_at`, `next_due_at`, `total_bookings`.
- Aggregate job daily:
  - For each tenant, compute cohorts: `active` (visited in last 30 days), `at_risk` (31–60), `churned` (> 60).
  - Write to `retention_metrics`: `{ tenant, date, active, at_risk, churned }`.

Endpoints:
- `GET /v1/tenants/{tenant}/customers/retention/summary` — returns counts and thresholds.
- `GET /v1/tenants/{tenant}/customers/retention/list?segment=at_risk&days=45` — list customers with phones for targeted promotions.
- `POST /v1/tenants/{tenant}/customers/retention/send` — trigger a promotion to the selected segment.

Scoring (simple):
- Score = weighted sum of recency, frequency, monetary (RFM) where monetary = avg price; store `score` in customers; expose in list.
